AWS DVA-C02 Study Guide

Comprehensive exam prep — key concepts, gotchas, and exam tips for the AWS Certified Developer Associate exam.

🔐 IAM — Identity & Access Management

Core Concepts

  • Users — physical person, one per human
  • Groups — collection of users (no nesting)
  • Roles — for AWS services / cross-account
  • Policies — JSON docs defining Allow/Deny

Policy Types

  • Identity-based — attached to user/group/role
  • Resource-based — attached to resource (S3, etc.)
  • Permissions Boundary — max permissions cap
  • SCP (Org) — max permissions for entire account
  • Session Policies — limit assumed role session
  • ACLs — cross-account, no JSON structure
Never use root account except for initial setup. Never store credentials in code.
Exam tip: --dry-run CLI flag tests permissions without executing. STS decode-authorization-message decodes encoded error messages.

IAM Federation

Large enterprises can use SAML 2.0 to integrate their own identity provider (Active Directory, etc.) with IAM. Users authenticate via their corporate IdP and get temporary AWS credentials via STS.

STS — Security Token Service

  • AssumeRole — assume a role in same or another account
  • AssumeRoleWithWebIdentity — for web identity federation (Cognito preferred)
  • AssumeRoleWithSAML — for SAML federation
  • GetSessionToken — for MFA

💻 EC2 — Elastic Compute Cloud

Instance Types

  • On-Demand — pay per second, no commitment
  • Reserved (1 or 3 yr) — up to 75% cheaper
  • Spot — up to 90% cheaper, can be reclaimed (2-min warning)
  • Dedicated Host — full physical server, BYOL
  • Dedicated Instance — hardware dedicated to you

Key Features

  • User Data — bootstrap script, runs once at first start as root
  • Metadata — http://169.254.169.254/latest/meta-data (inside instance only)
  • AMI — region-specific custom images
  • T2/T3 Burstable — uses CPU credits; T2 Unlimited = pay for extra bursts
Exam: EC2 metadata URL returns IAM role name but NOT the IAM policy. Memory metrics require CloudWatch Agent (not default).

💾 EBS — Elastic Block Store

TypeUse CaseBoot?

Key facts: EBS is locked to an AZ. Attach to only ONE instance at a time (except io1/io2 Multi-Attach). Snapshots are needed to move across AZs. Encryption uses KMS (AES-256).

Encryption

To encrypt an unencrypted volume: snapshot → copy snapshot with encryption enabled → restore from encrypted snapshot → migrate app → delete old DB.

⚖️ ELB & ASG

Load Balancer Types

  • ALB (Layer 7) — HTTP/HTTPS, path/host routing, WebSocket, containers. Client IP in X-Forwarded-For
  • NLB (Layer 4) — TCP/UDP, millions req/s, static IP, ~100ms latency. Sees client IP directly
  • CLB (Legacy) — avoid for new apps

ASG Key Points

  • Scale out/in based on CloudWatch alarms or custom metrics
  • Launch Configuration / Launch Template defines instance config
  • IAM roles on ASG are assigned to all EC2 instances
  • ASG is free — pay for underlying EC2s
  • ASG replaces unhealthy instances automatically
503 = LB at capacity or no registered targets. 504 = gateway timeout (backend issue).

🗄️ RDS & Aurora

RDS Features

  • Managed: patching, backups, monitoring
  • Read Replicas — async replication, up to 15 (MySQL/Postgres/MariaDB), 5 (Oracle/SQL Server)
  • Multi-AZ — sync replication, automatic failover, same DNS
  • Backups: automated (7-35 days) + manual snapshots
  • Encryption at rest: KMS AES-256
  • Cannot SSH into RDS instances
Read Replicas = performance (async). Multi-AZ = availability/DR (sync). They are different features.

Aurora

  • AWS proprietary, MySQL/Postgres compatible
  • 5x faster than MySQL RDS, 3x faster than Postgres RDS
  • Storage auto-grows in 10GB increments up to 128TB
  • Up to 15 read replicas, sub-10ms replica lag
  • Instantaneous failover (HA native)
  • Aurora Serverless — auto-scaling, pay per second
  • Global Aurora — 1 primary + up to 5 secondary regions, <1s lag
Exam: RDS 1040 "Too Many Connections" → enable RDS Proxy (connection pooling). Enhanced Monitoring = agent on instance. Default Monitoring = hypervisor level.

Caching Strategies

Lazy Loading (Cache-Aside)

  • Read from cache → miss → read DB → write to cache
  • ✅ Only caches requested data
  • ❌ Cache miss = 3 round trips; stale data possible

Write-Through

  • Write to DB → also write to cache
  • ✅ Data never stale
  • ❌ Cache churn; missing data until first write
TTL — set TTL to avoid stale data. Too many evictions → scale up/out.

🪣 S3 — Simple Storage Service

Basics

  • Globally unique bucket names, region-specific
  • Max object size: 5TB; use multipart upload for >100MB
  • 3500 PUT/s, 5500 GET/s per prefix
  • Versioning at bucket level

Encryption Methods

  • SSE-S3 — AWS managed keys, header: x-amz-server-side-encryption: AES256
  • SSE-KMS — KMS managed, header: aws:kms, audit trail
  • SSE-C — customer managed keys, HTTPS required, key in every request
  • Client-Side — encrypt before upload

S3 Security

  • Bucket Policies — JSON, resource-based, cross-account access
  • ACLs — finer grain per object
  • VPC Endpoints — access S3 without internet from VPC
  • MFA Delete — required for versioned bucket deletes
  • Pre-signed URLs — time-limited access to private objects

S3 Performance & Features

  • Transfer Acceleration — uses CloudFront edge locations; bucket name must be DNS-compliant (no periods)
  • S3 Object Lambda — transform data on-the-fly as it's returned (e.g., redact PII)
  • CORS — must enable if requesting from another S3 bucket/domain
  • Event Notifications → SQS, SNS, Lambda
  • Cross-Region Replication — requires versioning enabled on both buckets
Consistency: S3 now provides strong read-after-write consistency for all operations (as of Dec 2020). Old notes about eventual consistency are outdated for the exam.
Exam: 403 on static website = bucket policy doesn't allow public reads. CORS needed for cross-origin requests.

🌐 VPC & Security Groups

VPC Basics

  • VPC per Account per Region; Subnets per VPC per AZ
  • Public subnets: LBs, static sites, public auth layers
  • Private subnets: app servers, databases
  • VPC Peering — connect VPCs (same or cross-account)
  • VPC Flow Logs — monitor traffic in/out

Security Groups vs NACLs

  • Security Groups — stateful; allow inbound = allow return outbound automatically
  • NACLs — stateless; must explicitly allow both inbound AND outbound
  • SG: all inbound blocked by default, all outbound allowed
  • Timeout error = SG issue; Connection refused = app issue

🛠️ CLI & SDK

CLI Key Points

  • aws configure — set credentials locally
  • aws configure --profile name — multiple accounts
  • EC2 instances: use IAM Roles, never store credentials on instance
  • aws sts decode-authorization-message — decode encoded errors (needs STS permission)
  • --dry-run — test permissions without executing

SDK Key Points

  • Default region if not specified: us-east-1
  • Credential chain: code env vars → ~/.aws/credentials → IAM Role
  • NEVER store credentials in code
  • Exponential Backoff — for HTTP 429 (throttling); built into SDK
  • AWS CLI uses Python SDK (boto3)

🔄 CI/CD Overview

CodeCommit

Private Git repos. Auth via SSH or HTTPS. IAM for authorization. Encrypted at rest (KMS). Deprecated for new accounts.

CodeBuild

Managed build service. Uses buildspec.yml. Docker-based. Pay per build minute. Can run locally with CodeBuild Agent.

CodeDeploy

Deploy to EC2/On-Premise/Lambda. Uses appspec.yml. Agent required on EC2. Supports In-Place, Blue/Green, Linear.

CodePipeline

Orchestrates the pipeline. Artifacts stored in S3. State changes → CloudWatch Events → SNS. IAM role must have correct permissions.

📦 CodeCommit

  • Auth: SSH keys (configured in IAM) or HTTPS credentials
  • Cross-account access: use IAM Role + STS AssumeRole (never share SSH keys)
  • Notifications: SNS/Lambda for branch deletion, pushes to master; CloudWatch Events for PR updates
Deprecated — cannot create new repos. Existing repos still work.

Deployment Types

  • In-Place — existing instances updated
  • Blue/Green — new ASG created, traffic shifted, old ASG deleted. EC2 only (not on-premise)
  • Linear — equal increments at equal intervals (e.g., 10% every 10 min)
  • Canary — X% then 100%
CodeDeploy does NOT provision resources. Agent must be running on EC2/on-premise. Not needed for Lambda deployments.

🔁 CodePipeline

  • Stages: Source → Build → Test → Deploy (sequential or parallel actions)
  • Artifacts stored in S3 between stages (versioning must be enabled)
  • State changes → CloudWatch Events → SNS notifications
  • Manual approval can be added at any stage
  • If pipeline fails: check IAM Service Role permissions + CloudTrail for API audit

λ Lambda

Limits

  • Memory: 128MB – 10GB
  • Timeout: max 15 minutes (900s)
  • Env variables: 4KB
  • /tmp disk: 512MB – 10GB
  • Concurrent executions: 1000/account (soft limit)
  • Deployment package: 50MB compressed, 250MB uncompressed
  • Container image: up to 10GB

Invocation Types

  • Sync — ALB, API Gateway, CloudFront, CLI/SDK. Error handling on client side.
  • Async — S3, SNS, CloudWatch Events. 3 retries (0min, 1min, 2min). DLQ for failures.
  • Event Source Mapping — SQS, Kinesis, DynamoDB Streams. Lambda polls the source.

Concurrency & Throttling

  • Reserved Concurrency — guarantees max concurrency for a function; prevents it from using unreserved pool
  • Provisioned Concurrency — pre-warms instances, eliminates cold starts
  • Throttle: Sync → 429 ThrottleError; Async → retry + DLQ
  • One function without reserved concurrency can consume all 1000 concurrent executions

Versions & Aliases

  • Working version = $LATEST (mutable)
  • Published versions are immutable (own ARN)
  • Aliases are mutable pointers to versions; support weighted routing (Blue/Green)
  • Aliases cannot reference other aliases

Lambda + CodeDeploy Traffic Shifting

  • Linear — e.g., Linear10PercentEvery3Minutes
  • Canary — e.g., Canary10Percent5Minutes
  • AllAtOnce — immediate

Event Source Mapping Details

SourceBehavior
SQSLong polling, batch 1-10, DLQ on queue (not Lambda), visibility timeout = 6x Lambda timeout
Kinesis / DynamoDB StreamsOne invocation per shard, in-order, up to 10 parallel batches per shard
SQS FIFOScales to number of message groups, in-order per group
Never have a Lambda call itself recursively — can cause runaway costs.
Best practice: Initialize DB connections outside the handler (reused across invocations). Use /tmp for temp files, S3 for persistence.

Lambda@Edge

Run Lambda functions at CloudFront edge locations. Can modify viewer/origin requests and responses. Use cases: A/B testing, auth, bot mitigation, image transformation.

Lambda in VPC

  • By default Lambda runs outside your VPC (can access internet)
  • Lambda in VPC: creates ENI, needs AWSLambdaVPCAccessExecutionRole
  • Lambda in VPC cannot access internet unless NAT Gateway is configured
  • DynamoDB accessible via VPC endpoint or public internet

🗃️ DynamoDB

Capacity Units

  • WCU: 1 write/s for items up to 1KB. Round up item size.
  • RCU (Strong): 1 read/s for items up to 4KB
  • RCU (Eventual): 2 reads/s for items up to 4KB
  • Transactions: 2x WCU/RCU

WCU/RCU Examples

  • 10 items/s × 2KB = 20 WCU
  • 6 items/s × 4.5KB → round to 5KB = 30 WCU
  • 10 strong reads/s × 4KB = 10 RCU
  • 16 eventual reads/s × 12KB = (16/2) × (12/4) = 24 RCU

Indexes

FeatureLSIGSI
CreatedAt table creation onlyAnytime
Partition KeySame as tableAny attribute
ConsistencyStrong or EventualEventual only
CapacityShares table WCU/RCUOwn WCU/RCU
Throttle impactCannot throttle main tableGSI throttle = main table throttle

Key APIs

  • PutItem — create or full replace
  • UpdateItem — partial update, atomic counters
  • DeleteItem — conditional delete
  • BatchWriteItem — up to 25 PutItem/DeleteItem
  • GetItem — by primary key
  • BatchGetItem — up to 100 items
  • Query — by partition key (efficient)
  • Scan — full table scan (expensive, avoid)
  • TransactWriteItems — all-or-nothing across tables

DAX — DynamoDB Accelerator

  • In-memory cache, microsecond latency, no code changes needed
  • Solves Hot Key problem
  • Default 5-min TTL, up to 10 nodes, Multi-AZ

DynamoDB Streams

  • Changes (create/update/delete) pushed to stream
  • 24-hour retention (fixed)
  • Stream types: KEYS_ONLY, NEW_IMAGE, OLD_IMAGE, NEW_AND_OLD_IMAGES
  • Lambda reads via event source mapping (synchronous)

TTL

Auto-delete items after expiry. No extra cost. Deletion happens within 48 hours. TTL column must be a Number (epoch timestamp). Deleted items also removed from GSI/LSI.

GSI throttle = main table throttle. If GSI WCU is too low, writes to the main table will be throttled even if main table WCU is fine.
Table cleanup: Drop table (fast/cheap) vs Scan+Delete (slow/expensive). Copy table: backup → restore to new table.

Endpoint Types

  • Edge-Optimized (default) — global, routes via CloudFront
  • Regional — same region clients
  • Private — VPC only via VPC endpoint

Integration Types

  • MOCK — no backend call
  • HTTP/AWS — with mapping templates (VTL)
  • AWS_PROXY (Lambda Proxy) — no mapping, raw pass-through
  • HTTP_PROXY — raw pass-through to HTTP backend

  • Default throttle: 10,000 req/s across all APIs (soft limit)

  • Max integration timeout: 29 seconds

HTTP API vs REST API

  • HTTP API — cheaper, lower latency, supports OIDC/OAuth2, no usage plans/API keys, no data mapping
  • REST API — full features, usage plans, API keys, mapping templates, no native OIDC/OAuth2

  • Execution Role — pull images from ECR, write logs to CloudWatch

📦 ECR — Elastic Container Registry

  • Managed Docker registry (private and public)
  • Integrated with ECS, EKS, Lambda
  • Encryption at rest (AES-256 or KMS), in transit (TLS)
  • Image scanning for vulnerabilities (basic on push, enhanced with Inspector)
  • Lifecycle policies — auto-expire old/untagged images

☁️ Fargate

  • Serverless containers — no EC2 management
  • Works with ECS and EKS
  • Pay per vCPU-second and GB-second
  • Ephemeral storage: 20GB default, up to 200GB
  • For persistent storage: mount EFS
  • Cannot SSH into Fargate tasks

📬 SQS — Simple Queue Service

Standard Queue

  • Unlimited throughput, unlimited messages
  • At-least-once delivery (duplicates possible)
  • Best-effort ordering
  • Max message size: 256KB
  • Retention: 4 days default, max 14 days

FIFO Queue

  • Exactly-once processing
  • Strict ordering
  • 300 msg/s (3000 with batching)
  • Queue name must end with .fifo
  • Deduplication: content-based (SHA-256) or MessageDeduplicationID

Key Concepts

  • Visibility Timeout — 0s–12h (default 30s). Message invisible to others while being processed. Use ChangeMessageVisibility to extend.
  • Dead Letter Queue (DLQ) — after MaximumReceives exceeded. Set retention to 14 days. DLQ type must match queue type (FIFO DLQ for FIFO queue).
  • Long Polling — WaitTimeSeconds 1–20s. Preferred over short polling. Reduces API calls.
  • Delay Queue — DelaySeconds up to 15 min. Message not visible immediately.
  • Extended Client — for messages >256KB, stores payload in S3, sends pointer in message.
Lambda + SQS: Set visibility timeout = 6x Lambda timeout. Set DLQ on the SQS queue, NOT on Lambda. Lambda deletes messages after successful processing.

📢 SNS — Simple Notification Service

  • Pub/Sub model — one topic, many subscribers
  • Up to 10M subscriptions/topic, 100K topics
  • Subscribers: SQS, HTTP/HTTPS, Lambda, Email, SMS, Mobile Push
  • Messages not persisted — if subscriber is down, message is lost
  • Encryption: in-flight (HTTPS), at-rest (KMS), client-side

SNS + SQS Fan-Out Pattern

Publish to SNS → multiple SQS queues subscribe → each queue processes independently. Decoupled, no data loss, can add subscribers later.

Fan-out is the key pattern: SNS → multiple SQS queues. This is a very common exam scenario.

🌊 Kinesis

Kinesis Data Streams

  • Real-time streaming, ordered per shard
  • Write: 1MB/s or 1000 msg/s per shard
  • Read: 2MB/s per shard
  • Data retention: 1–365 days
  • Data can be replayed
  • Must provision shards

Kinesis Firehose

  • Near real-time (60s latency)
  • Fully managed, auto-scaling
  • Destinations: S3, Redshift, ElasticSearch, Splunk
  • Pay for data volume
  • No data storage/replay

📊 CloudWatch

Metrics

  • EC2 default: every 5 min. Detailed monitoring: every 1 min (cost)
  • Memory usage NOT included by default — must install CloudWatch Agent
  • Custom metrics: PutMetricData API
  • Standard resolution: 1 min. High resolution: 1 sec (StorageResolution)

Logs

  • Log Groups → Log Streams
  • Set expiration policy at Log Group level
  • Encryption with KMS at Group level
  • Export to S3 (batch) or stream to ElasticSearch
  • Lambda logs automatically go to CloudWatch (if IAM policy allows)

Alarms

  • States: OK, INSUFFICIENT_DATA, ALARM
  • Actions: Auto Scaling, EC2 actions, SNS notifications
  • High-resolution alarms: 10s or 30s periods

CloudWatch Events / EventBridge

  • Schedule: cron jobs (serverless CRON)
  • Event Pattern: react to AWS service events (CodePipeline state change, etc.)
  • Targets: Lambda, SQS, SNS, Kinesis

🔍 X-Ray

  • Distributed tracing for microservices
  • Segments → Sub-segments → Traces
  • Annotations (key/value, up to 50) — indexed, searchable
  • Metadata — not indexed, store any data

How to Enable

  1. Import X-Ray SDK in code (Java, Python, Go, Node.js, .NET)
  2. Install X-Ray Daemon on EC2 (or enable X-Ray integration on Lambda)
  3. IAM role must have write permissions to X-Ray

📋 CloudTrail

  • Enabled by default — records all API calls (Console, SDK, CLI, services)
  • Governance, compliance, and audit
  • Logs can be sent to CloudWatch Logs or S3
  • If a resource is deleted unexpectedly → check CloudTrail first
  • Can be used with Config for deeper resource change insight

⚙️ AWS Config

  • Compliance checks for AWS resources
  • Periodic or triggered evaluations
  • Requires IAM Role: read-only to resources, write to S3, publish to SNS
  • Use with CloudTrail for deeper audit insight

📄 CloudFormation

Template Sections

  • Resources — MANDATORY. AWS resources to create.
  • Parameters — dynamic inputs (!Ref)
  • Mappings — static variables (!FindInMap)
  • Outputs — export values for cross-stack reference
  • Conditions — conditional resource creation

Intrinsic Functions

  • !Ref — reference parameter or resource
  • !GetAtt — get attribute of resource
  • !FindInMap — lookup in Mappings
  • !ImportValue — import from another stack's Output
  • !Sub — string substitution
  • !Join — join strings with delimiter
  • !If / !And / !Or / !Not / !Equals — conditions
Cannot delete a stack if its Outputs are referenced by another stack. Templates must be re-uploaded to S3 (cannot edit in place). Deleting a stack deletes ALL resources it created.
Rollback: Stack creation failure → everything rolls back by default. Stack update failure → rolls back to last known working state.

Lambda in CloudFormation

Inline code (NodeJS/Python only, no dependencies) using ZipFile. For complex functions: reference S3 bucket/key. Must update S3ObjectVersion to trigger CloudFormation update.

  • Docker: add Dockerrun.aws.json to app directory
Optimization: Package dependencies with source code (not resolved on each EC2 at deploy time) to speed up deployments.

🔑 KMS — Key Management Service

  • Managed encryption keys for AWS services
  • Automatic key rotation: symmetric keys only (not asymmetric, not customer-imported keys)
  • Used by: S3 (SSE-KMS), EBS, RDS, DynamoDB, Lambda env vars, CloudWatch Logs, etc.
  • CMK (Customer Master Key): AWS managed or customer managed
  • Envelope encryption: data key encrypts data, CMK encrypts data key
Automatic rotation only works for symmetric CMKs. Asymmetric keys, imported keys, and customer-managed keys require manual rotation.

🌍 CloudFront

  • CDN — caches content at edge locations globally
  • Force SSL: set Viewer Protocol Policy to "Redirect HTTP to HTTPS" or "HTTPS only"
  • Geo-restriction: whitelist/blacklist countries
  • Custom headers: CloudFront-Viewer-Country, timezone, etc.
  • Origin types: S3, ALB, EC2, HTTP endpoint
  • Signed URLs / Signed Cookies — restrict access to premium content
  • OAI (Origin Access Identity) — restrict S3 access to CloudFront only

⚡ Exam Quick Reference

Common Error Codes

  • 400 — Bad Request (client error)
  • 403 — Forbidden (IAM/bucket policy)
  • 429 — Too Many Requests (throttling)
  • 500 — Internal Server Error
  • 502 — Bad Gateway (incompatible backend output)
  • 503 — Service Unavailable (LB no targets)
  • 504 — Gateway Timeout (backend timeout)

Default Regions / Limits

  • SDK default region: us-east-1
  • Lambda concurrent executions: 1000/account
  • Lambda timeout: 15 min max
  • API Gateway timeout: 29 seconds
  • SQS visibility timeout: 30s default, 12h max
  • SQS message retention: 4 days default, 14 days max
  • DynamoDB item max size: 400KB
  • DynamoDB Streams retention: 24 hours (fixed)

Key "Always" Rules

  • Never store credentials in code
  • Use IAM Roles for EC2/Lambda (not access keys)
  • Resource deleted unexpectedly → check CloudTrail
  • Performance issue → check X-Ray
  • Compliance check → AWS Config
  • Secrets needing rotation → Secrets Manager
  • Static config/params → SSM Parameter Store

Encryption Headers (S3)

  • SSE-S3: x-amz-server-side-encryption: AES256
  • SSE-KMS: x-amz-server-side-encryption: aws:kms
  • SSE-C: key in every request header, HTTPS required
